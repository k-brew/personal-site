    One escalated issue that I worked on was requested when it was reported that the for the dynamic documentation generated from the patient’s encounter data that contains follow-up instructions, a summary of visit, and other information pertaining to the nature of their visit was appearing blank for all patients being discharged from the emergency department at the hospital or being transferred to another unit/department within the hospital network.
    /
    For many reasons, it was important to determine what was causing the issue and how it could be resolved in a timely manner. By the end troubleshooting the issue, I discovered that the issue was caused following a change made by the client’s local IT to a specific discharge template that was intended to be a local change, but instead affected the entire organization. In technical terms, the change resulted in a break to the logic of the backend workflow where the code that automatically execute queries for the relevant information specific to the individual and characteristics of their encounter that populates the discharge documentation and the tokens that pull specific sets of information (depending on their configuration) were searching for data and instructions from two different sources instead of one, thus creating a computational-impossibility (in the context of the workflow).

    An outline of my workflow for the issue mentioned above is outlined as follows:

    • Check to see if issue is occurring in the live/production domain, the testing/training domain, or both. Since the issue is occurring in both domains, the resolution to the issue can be implemented and tested in the testing/training domain to avoid potential issues in live workflow. Continue troubleshooting process.
    • Look at configurations of the depart process for all locations experiencing the reported issue and check for obvious errors in depart template at a glance based on solutions from past similar situations.
      • Ensure depart/discharge process is configured for unit
      • Ensure the necessary settings in place for tracking groups affected by issue
      • Ensure tracking group assignments for affected units exist
    • After confirming all of the above, open text editor to inspect HTML of template for spelling errors, improper usage of syntax, or other errors resulting from updates to software or workflow. Verified that no such errors exist.
    • Open two different internal applications
      • First application for referencing core code sets, unique identifiers, and configuration metadata for different tracking groups and locations
      • Second application to writing and executing queries based on returned results from searches of core code set above to pull data from global and local databases
    • Search for relevant code sets and run queries to obtain the following infomration
      • Code values for locations experiencing issues
      • Code values for tracking groups experiencing issues
      • Code values for tracking group location assignments relationships
    • Merge results into single file and export .csv to computer
    • Open file and apply different filters to look for possible discrepancies. The issue has been found!
      • Assignment locations is a configuration option that creates an association between units and tracking groups. There had been a duplicate entry for one of these relationships that was causing a logical error in the workflow. Normally, the events that are triggered when a patient is transitions to the depart section of the clinical workflow look to a specific set of instructions/procedures depending on the tracking group. Instead what was happening was that the system was attempting to pull in directions for the workflow from two different sources, which is computationally impossible
    • Lastly, summarize findings and notify client of issue and what caused issue (to avoid future similar occurrences) and ask for verification of change in testing domain prior to production implementation. Ask for validation again after implementing into production domain and monitor for remainder of day to ensure proper functionality.
